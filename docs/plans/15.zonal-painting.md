# 15. Zonal Painting Implementation Plan

## 1. Problem Statement

The user wants to confine painting operations to a specific, user-defined rectangular area of the canvas, which we will call a "zone". All drawing logic, including coordinate calculations for colors and text parameters, should be relative to the zone's boundaries, treating the zone's top-left corner as `(0,0)`. This allows for more targeted and controlled creative compositions.

## 2. Requirements

### Functional Requirements

1.  **Zone Definition:** Users must be able to define a single, rectangular zone by clicking and dragging the mouse on the canvas.
2.  **Zone Lifecycle:**
    *   **Creation:** A "Define Zone" button puts the app into a mode where the next mouse drag creates a zone.
    *   **Activation/Deactivation:** Once a zone is defined, a toggle button ("Deactivate Zone" / "Activate Zone") allows the user to switch between painting inside the zone and painting on the main canvas, without losing the zone's content.
    *   **Clearing:** A "Clear Zone" button will permanently remove the defined zone and its contents.
3.  **Visual Feedback:**
    *   During zone definition (mouse drag), a dashed-line rectangle must indicate the selected area.
    *   When a zone is defined, it should have a persistent border. The border style should change to indicate whether the zone is currently active for painting (e.g., solid line) or inactive (e.g., dashed line). This border is for visualization only and should not be saved.
4.  **Confined Painting:** All subsequent drawing operations (from any drawing mode) must be rendered exclusively within the active zone.
5.  **Coordinate Translation:** All position-dependent calculations (e.g., for text size, color, radius) must be relative to the active zone's top-left corner, not the main canvas's.
6.  **Single Zone:** The initial implementation will support only one active zone at a time. Defining a new zone will replace the existing one.
7.  **Undo/Redo:** The existing undo functionality should work as-is, continuing to save and restore the state of the entire main canvas.

### Non-Functional Requirements

1.  **Performance:** The introduction of a zonal painting layer should not significantly degrade rendering performance.
2.  **Integration:** The feature must integrate cleanly with the existing drawing mode architecture (`grid-painter`, `circle-painter`, etc.).
3.  **State Management:** Zone state should be managed locally within the p5.js sketch instance, not in the global Vuex store.

## 3. Technical Approach

The core of the implementation will rely on the `p5.Graphics` object, often referred to as an off-screen graphics buffer.

1.  **Zone Representation:** The active zone will be represented by a `p5.Graphics` object. This object will have the dimensions of the user-defined rectangle. We will also store its top-left `(x, y)` position on the main canvas.
2.  **Drawing Workflow:**
    *   When a zone is active, all drawing mode functions (`drawGrid`, `drawCircle`, etc.) will be passed the zone's `p5.Graphics` object as the rendering target.
    *   These functions will execute their drawing commands on this off-screen buffer. The coordinate system is naturally handled, as `(0,0)` in the buffer corresponds to its top-left corner.
    *   In the main `p.draw()` loop, after the main canvas is cleared/updated, we will render the zone's `p5.Graphics` buffer onto the main canvas at its stored `(x, y)` position using `p.image()`.
3.  **UI and State:**
    *   A new state property, `activeZone`, will be added to the `sketch.js` instance. It will hold the zone's state (`{ graphics, x, y, width, height }`) or be `null` if no zone is defined.
    *   A new boolean state, `isZonePaintingActive`, will control whether active drawing is directed to the zone or the main canvas.
    *   The GUI will feature conditional buttons:
        *   If no zone is defined: show "Define Zone".
        *   If a zone is defined: show "Clear Zone" and a toggle button for "Activate Zone" / "Deactivate Zone".
    *   Mouse events (`mousePressed`, `mouseDragged`, `mouseReleased`) in `sketch.js` will be updated to handle zone definition when in the appropriate mode.

## 4. Implementation Steps

1.  **`sketch.js` Modifications:**
    *   Add new state variables to the sketch instance:
        *   `isDefiningZone = false`
        *   `isZonePaintingActive = false`
        *   `zoneStartPos = null` (to track the initial click)
        *   `activeZone = null` (to store `{ graphics, x, y, width, height }`)
    *   Update `p.draw()`:
        *   If `isDefiningZone` and `zoneStartPos` are set, draw the dashed-line preview rectangle on the main canvas.
        *   If `activeZone` exists, draw its border on the main canvas, changing the style based on `isZonePaintingActive`.
        *   After all primary drawing is complete, if `activeZone` exists, render `activeZone.graphics` to the main canvas using `p.image(activeZone.graphics, activeZone.x, activeZone.y)`.
    *   Update mouse handlers:
        *   `mousePressed`: If `isDefiningZone`, store the mouse position in `zoneStartPos`. Otherwise, proceed with normal drawing logic.
        *   `mouseDragged`: If `isDefiningZone`, update the preview. Otherwise, proceed with normal drawing.
        *   `mouseReleased`: If `isDefiningZone`, create the `p5.Graphics` object based on the final rectangle, store it in `activeZone`, set `isZonePaintingActive` to `true`, and reset `isDefiningZone` and `zoneStartPos`.

2.  **GUI Component (`PolychromeText.vue` or similar):**
    *   Add conditional buttons to the main control panel.
    *   If `activeZone` is `null`:
        *   Show a "Define Zone" button. This sets `isDefiningZone` to `true`.
    *   If `activeZone` is not `null`:
        *   Show a "Clear Zone" button. This sets `activeZone` to `null` and `isZonePaintingActive` to `false`.
        *   Show a toggle button:
            *   Label: "Deactivate Zone" if `isZonePaintingActive` is true, "Activate Zone" if false.
            *   Handler: Toggles the `isZonePaintingActive` boolean.

3.  **Drawing Mode Integration:**
    *   Refactor the core drawing dispatch logic (likely in `sketch.js` where `renderLayers` or similar is called).
    *   The function that calls the drawing mode painters must now determine the correct rendering target.
        *   `const target = activeZone && isZonePaintingActive ? activeZone.graphics : p;`
    *   Modify the signatures of the drawing functions in `grid-painter.js`, `circle-painter.js`, and `rowcol-painter.js` to accept the target graphics context as an argument.
        *   Example: `drawGrid(params)` becomes `drawGrid(params, target)`.
    *   Update all calls within these painters from `p.rect(...)` to `target.rect(...)`. The original `p` instance may still be needed for accessing p5 constants or utilities (`p.HSB`, `p.random`, etc.). The factory pattern for these modules should be updated to pass both `p` and the `target`. A better approach would be to pass a single `context` object containing `{p, target}`.

4.  **Coordinate Calculation:**
    *   The drawing modes need to be passed the correct mouse coordinates relative to the zone.
    *   When a zone is active and `isZonePaintingActive` is true, the mouse coordinates passed to the drawing functions must be translated: `(mouseX - activeZone.x, mouseY - activeZone.y)`. This logic should reside in the main sketch's mouse event handlers before calling the drawing functions.

## 5. Testing Strategy

1.  **Unit Tests (Vitest):**
    *   Test the coordinate translation logic to ensure mouse positions are correctly mapped to the zone's local coordinates.
    *   Test the logic for creating and clearing the `activeZone` state object.
2.  **E2E Tests (Playwright):**
    *   **Test Case 1: Zone Definition and Visualization.**
        *   Click "Define Zone" button.
        *   Drag the mouse on the canvas.
        *   Verify the dashed preview rectangle appears.
        *   Release the mouse.
        *   Verify a persistent border appears around the new zone.
        *   Verify the button text changes to "Clear Zone".
    *   **Test Case 2: Confined Painting.**
        *   Define a zone.
        *   Select a drawing mode (e.g., Grid).
        *   Perform a drawing action inside the zone.
        *   Verify the drawing appears only within the zone's boundaries.
        *   Perform a drawing action outside the zone.
        *   Verify no new drawing appears on the canvas.
    *   **Test Case 3: Coordinate Relativity.**
        *   Define a zone in the bottom-right quadrant of the canvas.
        *   Use a drawing mode where color or size depends on `(x,y)` position.
        *   Click in the top-left corner of the zone.
        *   Verify the output corresponds to `(0,0)` parameters (e.g., the color/size expected at the canvas's top-left corner in normal mode).
    *   **Test Case 4: Clearing the Zone.**
        *   Define a zone and draw in it.
        *   Click "Clear Zone".
        *   Verify the zone border disappears.
        *   Perform a drawing action.
        *   Verify the drawing now occurs on the main canvas as normal.

## 6. Risks & Mitigation

*   **Performance Overhead:** `p.createGraphics` is generally efficient, but creating very large off-screen buffers could add memory pressure.
    *   **Mitigation:** The impact is expected to be minimal since we are only supporting one zone. We will monitor performance during testing.
*   **Refactoring Complexity:** Modifying the signatures of all drawing functions is a significant change.
    *   **Mitigation:** The change is straightforward (`p` -> `target`). We will perform this refactoring systematically, one drawing mode at a time, with tests to ensure no regressions.
*   **Event Handling Logic:** The mouse event handlers in `sketch.js` will become more complex with the addition of the "zone definition" state.
    *   **Mitigation:** We will use clear state flags (`isDefiningZone`) and well-structured conditional logic to manage the different modes of interaction.

## 7. Dependencies

*   **p5.js:** Core dependency for `createGraphics`.
*   ~~**Vue.js/Nuxt:** For UI button and event binding.~~
*.  **gui.js** For UI buttons and event binding.
*   **Existing Drawing Mode Modules:** These will be the primary modules affected by the refactoring.
